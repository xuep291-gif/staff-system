# 高校收费管理系统 财务端需求规格说明书

**版本：** V1.0  
**日期：** 2026-05-30  
**状态：** 初稿  
**依据：** 《高校收费管理系统需求规格说明书_V3.0》  
**适用范围：** 财务人员、迎新工作人员所使用的移动端（小程序/H5）及后台管理端（PC Web）

## 一、文档说明

### 1.1 编写目的

本文档面向高校收费管理场景中**财务端**（含财务人员移动端、后台管理端），系统化描述财务相关业务需求、功能模块、角色权限、数据字段及业务规则。供财务端开发团队作为设计、开发与验收的依据。

### 1.2 财务端角色定义

| 角色 | 终端 | 权限范围 |
|------|------|----------|
| 财务人员 | 小程序 / H5 + PC Web | 收费确认、退费操作（含补差退款）、票据管理、催缴通知、预缴批扣、数据导入导出、统计查询 |
| 迎新工作人员 | 小程序 / H5 | 现场缴费核验（扫码/输入学号查询缴费状态，决定放行）、现场收款登记 |
| 系统管理员 | PC Web | 收费配置、全部审批操作、系统配置管理 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 支付订单 | 单次支付请求单，关联账单/明细；状态：待支付 / 支付成功 / 已关闭 / 支付失败 |
| 缴费记录 | 资金入账凭证，一条支付成功订单对应一条或多条缴费记录（按明细拆分时） |
| 合并缴费组 | 合并多笔账单一次支付时，系统记录的账单与支付金额映射关系集合 |
| 批扣 | 银行批量代扣，学校将应缴名单提交银行，银行统一扣款 |
| 预缴 | 学生在账单生成前提前缴纳的费用，后续自动抵扣 |
| 预缴余额 | 学生当前可用于抵扣的预缴金额 |
| 电子票据 | 支付成功后系统自动生成的电子收据，具有法律效力 |
| 补差订单 | 因宿舍房型变更产生的费用差额订单（补缴或退款） |
| 助学金抵扣 | 审批通过且发放方式为账单抵扣时，自动冲抵学费账单（增加优惠金额） |
| 助学贷款冲抵 | 贷款审批通过后自动冲抵学费账单 |

## 二、核心业务闭环（财务视角）

### 2.1 缴费闭环

**主路径（在线支付）：**

```
配置收费标准 / 个人调整
    ↓
身份绑定或学年切换 → 生成账单（状态：未缴）
    ↓
学生选择账单或明细项 → 系统按优先级规则分配金额 → 创建支付订单（待支付，30 分钟有效）
    ↓
调用第三方支付 → 用户完成付款
    ↓
支付回调（验签 + 幂等，以支付订单号为去重键）
    ↓
写入缴费记录 → 通过账单服务（分布式锁，账单 ID 粒度）串行更新已缴金额 → 重算账单状态
    ↓
生成电子票据 → 发送缴费成功通知
```

**财务侧关联入口：**

| 路径 | 财务操作 |
|------|----------|
| 线下收款 | 财务确认收款、登记凭证 |
| 银行批扣 | 导出批扣文件 → 提交银行 → 导入批扣结果 |
| 预缴抵扣 | 财务导入预缴数据 |
| 补差补缴 | 推送后等待学生缴费 |

**支付订单状态机：**

| 状态 | 说明 | 可转移至 |
|------|------|----------|
| 待支付 | 已创建，等待用户付款 | 支付成功 / 已关闭（超时） / 支付失败 |
| 支付成功 | 渠道回调确认 | 终态（允许发起退费） |
| 已关闭 | 超过 30 分钟未支付，自动关闭 | 终态（可重新创建新订单） |
| 支付失败 | 渠道明确返回失败 | 终态（可重新创建新订单） |

**部分缴费金额分配规则（D-02 已定）：**

学生选择部分缴费（缴纳金额 < 账单应缴总额）时，系统按收费项目优先级从高到低依次填满各项，直至金额用尽：

1. 按收费项目"收费优先级"字段升序排列（数值越小越优先）；
2. 当前项目"已缴金额"累加至"应缴金额"为止，剩余金额流向下一项；
3. 必缴项目（`是否必缴=1`）优先纳入，但仍按优先级排序；
4. 缴费确认页实时展示"本次分配明细预览"，学生确认后方可提交支付；
5. 任一项目已缴金额不得超过该项应缴金额。

**合并缴费与退费（D-03 已定）：**

- 合并缴费时系统创建"合并缴费组"，持久化记录各账单 ID 与本次支付中对应金额的映射；
- 退费申请时，若原缴费来自合并缴费，退费申请关联对应的合并缴费组 ID；
- 退款接口调用金额以该账单在本次合并缴费中的映射金额为上限；
- 支持对合并缴费中的单笔账单单独发起退费，其余账单不受影响。

**账单金额演算规则：**

- `已缴金额` = Σ 成功缴费记录金额 + Σ 抵扣类冲抵金额 − Σ 成功退费回退金额
- `欠费金额（展示）` = `应缴总金额` − `已缴金额` − `优惠金额`
- 账单状态：`已缴=0` → **未缴**；`0 < 已缴 < 应缴` → **部分缴**；`已缴 ≥ 应缴` → **已缴**

**并发写入保护：**

`已缴金额`字段涉及多路并发写入（支付回调、预缴抵扣、助学金冲抵、助学贷款冲抵、退费回退），所有修改必须通过账单服务的分布式锁（按账单 ID 粒度）串行执行，禁止直接 UPDATE。所有写入以"账单 ID + 来源类型 + 业务流水号"为幂等键去重。

**闭环异常与补偿：**

| 异常场景 | 处理策略 |
|----------|----------|
| 回调延迟或重复 | 以支付订单号为幂等键；重复回调不重复记账 |
| 支付成功但未生成票据 | 对账任务补生成电子票据 |
| 支付成功但账单未更新 | 对账任务按订单补写缴费记录并刷新账单 |
| 部分缴费后停止 | 账单保持部分缴；催缴模块跟进欠费项 |

### 2.2 退费闭环

**主路径（在线支付原路退回）：**

```
学生发起退费申请（填写退费原因）
    ↓
系统校验（见校验清单）
    ↓
创建退费申请（状态：待财务审核）
    ↓
学生端展示：「退款申请已提交，审核中」
    ↓
财务人员在职工端/PC端确认退费 → 系统调用第三方退款接口（状态：处理中）
    ↓
退款回调（验签 + 幂等）
    ↓
退费成功 → 回退账单/明细已缴金额 → 重算账单状态
    ↓
处理关联电子票据（全额退：作废原票；部分退：保留原票并记录已退金额）
    ↓
发送退费结果通知（成功/失败）
```

**退费状态机：**

| 状态 | 说明 | 学生端展示文案 | 财务端可见 |
|------|------|----------------|-----------|
| 待财务审核 | 已提交申请，财务未确认 | 退款申请已提交，审核中 | 待处理列表，可操作确认 |
| 处理中 | 财务已确认，已提交第三方 | 退款处理中，预计 3-5 个工作日到账 | 处理中列表 |
| 成功 | 退款到账，账单已回退 | 退款成功，已原路返回 | 已完结列表 |
| 失败 | 渠道失败，需人工介入 | 退款申请已提交，审核中（统一展示） | 展示失败原因，支持重新发起 |

> **退费失败对学生的可见程度（D-05 已定）：** 无论失败原因（系统超时、账户异常等），学生端统一展示"退款申请已提交，审核中"。失败详情仅对财务人员可见，财务人工处理后重推。

**退费校验清单：**

| 校验项 | 不通过时行为 |
|--------|-------------|
| 原支付状态 = 支付成功 | 拒绝申请，提示"该订单不支持退费" |
| 支付日期 = 当日（在线支付） | 拒绝申请，提示"退费仅支持当日申请" |
| 退费金额 ≤ 可退余额 | 截断为可退上限，展示最大可退金额 |
| 同一缴费记录无重复全额退 | 拒绝申请 |
| 来自合并缴费 | 须关联合并缴费组 ID，退款金额 ≤ 该账单在组内的映射金额 |
| 涉及助学金抵扣 | 可退上限 = 原现金支付金额（不含助学金抵扣部分） |

**退费入口展示规则：**
- 退费入口仅在当日支付成功的订单上可操作
- 过了当日（服务器时间所属自然日），入口变为置灰禁用状态并展示"仅支持当日申请退费"
- 前后端均以服务器时间为准

**退费申请流程（D-04 已定）：学生提交 → 财务人工确认 → 系统执行退款**

- 学生提交后状态立即变为"待财务审核"
- 财务人员在职工端/PC端退费处理页看到该申请，确认后系统调用退款接口
- 退款发起后状态变为"处理中"
- 回调成功后变为"成功"，同步回退账单，发送通知
- 退款失败时财务端展示失败原因，支持重新发起；学生端保持"退款申请已提交，审核中"展示

**可退金额计算规则（含助学金场景）：**

- 可退金额 = min（原现金支付金额，账单已缴金额 − 优惠金额）
- 助学金抵扣部分（计入优惠金额）不可通过退费流程退回；如需退回，走助学金管理模块独立流程
- 退费申请页展示："本次最高可退金额：¥XX（不含助学金抵扣部分 ¥XX）"

**线下收费退费分支：**
- 不调用第三方接口
- 财务在后台登记退款方式（现金/转账等）并确认后，同样写入退费记录、回退账单、处理票据

### 2.3 宿舍补差闭环

**触发条件：** 宿舍分配结果变更导致住宿费标准变化。

**参与实体：** 宿舍分配记录 → 补差订单 → 账单/账单明细 → 缴费或退费闭环 → 通知

**主路径：**

```
宿舍变更（学生申请换房审批通过 / 智能分配 / 管理员手动调换）
    ↓
快照原宿舍、新宿舍、原房型费用、新房型费用
差额 = 新费用 − 原费用（正数=补缴，负数=应退）
    ↓
生成补差订单（状态：待推送）
补缴截止日期 = 补差订单生成时间 + 后台配置宽限天数
    ↓
    ├─ 差额 > 0（补缴）→ 追加到当前学年账单明细 → 学生缴费 → 补差订单（已结清）
    │
    └─ 差额 < 0（应退）→ 生成独立待退金额记录
          财务在职工端"补差退款"列表确认后执行退款
          回退账单已缴金额 → 补差订单（已结清）
```

**财务侧职责：**
- **补差补缴：** 无需财务介入，学生自行缴费
- **补差退款：** 财务在职工端「补差退款处理」列表确认退款 → 系统执行退款 → 回退账单

**补差补缴截止日期（D-06 已定）：**
- 截止日期 = 补差订单生成时间 + 后台配置的宽限天数（默认 7 天）
- 选宿批次配置页新增"补差补缴宽限天数"字段（INT，必填，默认 7）
- 距截止 ≤ 3 天时学生端以警告色高亮

**补差订单状态机：**

| 状态 | 说明 |
|------|------|
| 待推送 | 已计算差额，未进入缴费系统 |
| 已推送 | 已写入账单调整或生成待退记录 |
| 待补缴 | 差额 > 0，学生尚未缴清 |
| 待退款 | 差额 < 0，退款尚未完成 |
| 已结清 | 补缴或退款均已完成 |
| 推送失败 | 对接异常，可人工重推 |

## 三、财务移动端（小程序 / H5）

### 3.1 功能页面清单

#### 3.1.1 缴费仪表盘（首页）

**功能说明：** 全校当日收费汇总，提供核心数据概览和快捷入口。

**展示数据：**
- 当日收费总额
- 当日笔数
- 未缴人数
- 各支付方式占比

**快捷入口：**
- 线下收款
- 查看待确认记录

#### 3.1.2 线下收款确认

**功能说明：** 查看待确认线下收款记录，执行"财务确认"操作。

**操作流程：**

```
学生线下缴费（现金/转账等）→ 上传凭证或迎新现场登记
    ↓
财务人员在「线下收款确认」列表查看待确认记录
    ↓
核实凭证 → 点击"确认收款"
    ↓
账单状态更新 → 生成电子票据 → 发送缴费通知
```

**数据字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| 记录编号 | VARCHAR(50) | 唯一标识 |
| 学生姓名 | VARCHAR(50) | |
| 学号 | VARCHAR(30) | |
| 缴费金额 | DECIMAL(10,2) | |
| 收款方式 | 枚举 | 现金/银行转账/微信/支付宝/其他 |
| 收款地点 | VARCHAR(100) | 迎新现场：收款台编号 |
| 提交时间 | DATETIME | |
| 确认状态 | 枚举 | 待确认/已确认/已作废 |
| 确认人 | VARCHAR(50) | 确认操作的财务人员姓名 |
| 确认时间 | DATETIME | |
| 收据编号 | VARCHAR(50) | 确认后自动生成 |

**业务规则：**
- 财务确认后账单状态立即更新
- 已确认的记录可作废（需管理员权限），作废后回退账单金额
- 作废记录不可恢复

#### 3.1.3 收款记录查询

**功能说明：** 按时间、学院、支付方式等多维度查询收款记录。

**筛选条件：**
- 时间范围
- 学院
- 支付方式（在线/线下/批扣/预缴抵扣）
- 收款状态

**列表展示：**
- 学生姓名、学号
- 缴费金额
- 支付方式
- 支付时间
- 账单编号

#### 3.1.4 退费处理

**功能说明：** 查看退费申请列表，执行退费确认，查看失败详情，对失败退费重新发起。

**Tab 分类：**

| Tab | 包含状态 | 说明 |
|-----|----------|------|
| 待财务审核 | 待财务审核 | 学生已申请，等待财务确认 |
| 处理中 | 处理中 | 已提交第三方退款接口 |
| 已完结 | 成功 / 失败 | 含成功和失败记录 |

**操作：**
- **确认退费：** 查看申请详情 → 核实信息 → 点击确认 → 系统调用退款接口
- **重新发起：** 退款失败时，财务可查看失败原因并重新发起
- **失败详情：** 仅财务端可见失败原因，学生端不暴露

**数据字段（详情页）：**

| 字段 | 说明 |
|------|------|
| 申请编号 | 格式：RFA{YYYYMMDD}{6 位流水号} |
| 学生信息 | 姓名、学号、学院、班级 |
| 原支付订单 | 订单编号、支付金额、支付方式、支付时间 |
| 退费金额 | 本次申请退费金额 |
| 可退上限 | 该订单的最大可退金额 |
| 退费原因 | 学生填写的退费原因 |
| 申请时间 | |
| 当前状态 | 待财务审核/处理中/成功/失败 |
| 失败原因 | 仅失败时展示，仅财务可见 |
| 关联合并缴费组 | 若来自合并缴费 |
| 原票据处理 | 全额退作废 / 部分退保留并记录已退金额 |

**业务规则：**
- 退费须由财务人员人工确认后方可执行
- 退款失败时支持财务重新发起，不限重试次数
- 失败记录保留完整审计日志

#### 3.1.5 补差退款处理

**功能说明：** 查看待处理的补差应退记录（差额 < 0 的补差订单），确认后执行退款。

**操作流程：**

```
筛选差额 < 0 且状态为"待退款"的补差订单
    ↓
财务查看详情 → 确认退款
    ↓
系统执行退款（不关联原缴费记录，不受"当日退费"限制）
    ↓
回退账单已缴金额 → 补差订单（已结清）→ 通知学生
```

**数据字段：**

| 字段 | 说明 |
|------|------|
| 补差订单号 | 格式：DIFF{YYYYMMDD}{流水号} |
| 学生信息 | 姓名、学号 |
| 原房型 | 原宿舍房型 |
| 新房型 | 新房型 |
| 差额 | 负数（应退金额） |
| 生成时间 | 补差订单生成时间 |
| 状态 | 待退款 / 已结清 |

#### 3.1.6 票据管理

**功能说明：** 查询票据列表，协助学生补打票据，作废错误票据。

**操作功能：**

| 功能 | 说明 |
|------|------|
| 票据查询 | 按学号、票据编号、时间范围查询 |

|------|------|----------|
| 微信模板消息 | 通过微信公众号/小程序发送 | 学生已关注并绑定 |
| 短信 | 通过短信服务商发送 | 学生手机号已录入 |
| 站内消息 | 系统内部消息，登录后可查看 | 学生已注册账号 |

**渠道优先级：** 微信模板消息 > 短信 > 站内消息

#### 4.10.2 通知场景清单（财务相关）

| 场景编码 | 触发时机 | 渠道 |
|----------|----------|------|
| FEE_PAYMENT_SUCCESS | 在线支付成功 / 线下收款确认 | 微信/短信/站内 |
| FEE_PAYMENT_RECEIPT | 学生申请退费并提交成功 | 站内（告知已受理） |
| FEE_REMINDER | 账单逾期，按催缴规则触发 | 短信（强制）+ 微信/站内 |
| FEE_REFUND_RESULT | 退费成功或失败 | 微信/短信/站内 |
| FEE_BATCH_DEDUCT_SUCCESS | 银行批扣成功 | 微信/短信/站内 |
| FEE_BATCH_DEDUCT_FAILED | 银行批扣失败 | 短信（强制）+ 微信/站内 |
| FEE_DIFF_ORDER | 补差订单生成并推送 | 微信/短信/站内 |
| FEE_DIFF_DEADLINE | 补差补缴截止前 3 天预警 | 微信/短信/站内 |
| PRECOLLECT_EXPIRE | 预缴余额到期前 30 天提醒 | 站内/短信 |

#### 4.10.3 业务规则

- 通知发送失败时自动重试（最多 3 次，间隔 5 分钟）
- 重试仍失败时记录失败原因，支持手动重发
- 通知发送记录保留 180 天
- 同一学生同一账单催缴通知 7 天内只发送一次
- 每日催缴发送时间限制 09:00–20:00
- 单次催缴任务最多发送 1000 条

## 六、非功能性需求

### 6.1 性能要求

| 场景 | 性能指标 |
|------|----------|
| 报表统计 | 常规报表响应时间 < 5 秒 |
| 批量账单生成 | 支持 10000 人以上批量生成，异步处理 |
| 并发支付 | 支持迎新高峰期 500+ 并发支付 |
| 账单写入 | 分布式锁保证串行；单账单锁等待超时 ≤ 3 秒，超时返回重试 |
| 线下收款录入 | 操作响应时间 < 1 秒 |

### 6.2 安全要求

- 支付渠道配置（商户号、密钥等）加密存储
- 支付回调需验证签名，防止伪造
- 学生身份信息（身份证号等）脱敏展示
- 所有操作记录审计日志，不可删除
- 退费操作须财务二次确认，防止误操作
- 票据补打需身份验证

### 6.3 可用性要求

- 系统可用性 ≥ 99.9%
- 迎新高峰期（每年 9 月）提前做好容量规划
- 支付回调处理需保证幂等性

### 6.4 数据规范

- 金额字段统一使用 DECIMAL(10,2)，单位：元
- 日期字段统一使用 ISO 8601 格式（YYYY-MM-DD）
- 时间字段统一使用 ISO 8601 格式（YYYY-MM-DD HH:MM:SS）
- 所有编号采用统一格式规范
- 服务端时间为权威时间来源，禁止依赖客户端本地时间
