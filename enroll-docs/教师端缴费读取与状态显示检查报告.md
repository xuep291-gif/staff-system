### 检查 4：教师端查询使用的 student_id 是否和学生端写入的 student_id 一致？

**结论：✅ 一致，都使用 `t_data_student.id`**

#### Java 后端写入侧（学生端支付）

[BillController.java:25](d:\study\backend\enroll-service-java\src\main\java\com\jfeat\enroll\controller\student\BillController.java#L25):
```java
private static final String STUDENT_ID_SQL = "(SELECT id FROM t_data_student WHERE student_no = ?)";
```

`savePaymentRecord` 用 `sid`（JWT 中的 student_no）调用 `STUDENT_ID_SQL` → 解析为 `t_data_student.id`。

写入 `t_data_fee_billing.student_id` 和 `t_data_fee_payment_record.student_id` 的值都是 `t_data_student.id`。

#### JS 后端读取侧（教师端）

[payments.ts:64-68](d:\study\backend\enroll-js\src\routes\staff\payments.ts#L64-L68):
```sql
LEFT JOIN t_data_fee_billing b ON s.id = b.student_id::bigint
```

[payments.ts:106](d:\study\backend\enroll-js\src\routes\staff\payments.ts#L106):
```sql
SELECT * FROM t_data_fee_billing WHERE student_id::text = ${String(s.id)}
```

两侧都使用 `t_data_student.id` 作为 `student_id`。✅

**需要注意的是**：Java 后端存的是 `student_id = (SELECT id FROM t_data_student WHERE student_no = ?)` — 这是数字型 bigint。JS 后端读的时候做了 `student_id::text` 转换后用 `s.id`（也是数字）的字符串形式比较。PostgreSQL 会自动做类型转换，所以 JOIN 能匹配上。

**结论：fee-home 无 fallback（但后端始终返回），student-detail 行项有 fallback，整体无 fallback**

#### 6.1 fee-home/index.vue

[line 243](d:\study\backend\迎新收费系统教师端\迎新收费系统教师端\enroll-app\src\pages\teacher\fee-home\index.vue#L243):

❌ fallback 直接写死 `'unpaid'`，没有根据 `paidAmount` 计算。例如如果后端返回 `paidAmount=5000, receivableAmount=5000` 但 `paymentStatus` 意外为空，仍然显示"未缴"。

#### 6.3 student-detail/index.vue（行项 billStatus）

[line 144](d:\study\backend\迎新收费系统教师端\迎新收费系统教师端\enroll-app\src\pages\teacher\student-detail\index.vue#L144):

✅ 正确。如果 `b.status` 为空，根据 `paidAmount` vs `receivableAmount` 计算：
- `paidAmount >= receivableAmount` → `paid`
- `paidAmount > 0` → `partial`
- 其他 → `unpaid`
