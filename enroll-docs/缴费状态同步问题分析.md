> 项目：迎新收费管理系统（教师端 + 学生端）  
> 模块：缴费业务闭环  
> 分析日期：2026-05-31  
> 范围：只分析，不修改代码

---

## 一、问题现象

1. **学生详情页应缴金额和学生端缴费金额对不上**
2. **学生端已经缴费后，教师端仍然显示"部分缴费/未缴"**
3. **明细项目中学费、住宿费等状态仍显示未缴**

### 关键观察

**所有"已缴费"状态仅存在于前端 localStorage 和 Pinia 内存中。** 后端的 `t_data_fee_billing.paid_amount` 字段从未被学生支付流程更新。教师端查询的是 `t_data_fee_billing`，因此永远看不到学生的支付。

## 五、根因分析（10 个问题逐一排查）

### 问题 1：应缴金额 totalAmount 来源不一致

| 端 | totalAmount 来源 | 值 |
|----|-----------------|-----|
| 学生端 `bill/index.vue` | `paySystem` 硬编码 | 8480（含住宿2000）或 6480（不住宿） |
| 学生端 `checkout/index.vue` | 硬编码 | 同上 |
| 学生端 `loadBillPrices()` | `GET /api/student/bill/list` → `bills[].amount` | **来自后端，但仅用于覆盖个别价格** |
| 教师端 `fee-home` | `GET /api/v1/payments/students` → `SUM(t_data_fee_billing.total_amount)` | **来自数据库** |
| 教师端 `student-detail` | `GET /api/v1/payments/students/:id` → `t_data_fee_billing.total_amount` | **来自数据库** |
| businessState.js `getFeeList()` | `FEE_ITEMS_CONFIG` 硬编码 SUM | 6900（5000+1200+320+200+180） |

**结论：学生端用硬编码价格（8480），教师端用数据库（取决于 t_data_fee_billing 中实际数据），businessState.js 用另一套硬编码（6900）。三者可能完全不同。**

**根因类型：前端硬编码价格**

---

### 问题 2：已缴金额 paidAmount 来源不一致

| 端 | paidAmount 来源 | 存储位置 |
|----|----------------|----------|
| 学生端 `bill/index.vue` | `updatePaymentState()` 读 `localStorage.paidBills` 累加 | 浏览器 localStorage |
| 学生端 `store` | `totalPaidAmount` ref | Pinia 内存 + localStorage |
| 教师端 | `SUM(t_data_fee_billing.paid_amount)` | PostgreSQL |
| 学生详情页 `student-detail` | `paymentRes.data.summary.paidAmount` | PostgreSQL `t_data_fee_billing` |

**结论：学生端已缴金额来自 localStorage，教师端来自数据库 t_data_fee_billing。学生支付后只写 localStorage 和 t_data_fee_payment_record，从未更新 t_data_fee_billing.paid_amount → 教师端永远看到 paidAmount = 0 或初始值。**

**根因类型：后端数据同步缺失**

---

### 问题 3：未缴金额 unpaidAmount 计算方式不同

| 端 | 计算方式 |
|----|---------|
| 学生端 | `totalPayNum - actualPay`（硬编码总额 - localStorage 已缴） |
| 教师端 | `MAX(0, total - paid)`（SQL 中 `t_data_fee_billing` 字段相减） |

**结论：两端使用完全不同的分子和分母。**

**根因类型：前端自行计算 + 后端数据不同步**

---

### 问题 4：缴费项目 items 的 paid/unpaid 状态是否和支付记录同步

**不同步。**

- **学生端**：项目级别的 paid/unpaid 通过 `localStorage.paidBills` 对象跟踪（`{ tuition: true, book: true, ... }`）。支付后 `checkout/index.vue:doPay()` 写入此对象。
- **教师端 `student-detail`**：`bills[]` 来自 `t_data_fee_billing` 表，每行的 `status` 字段。后端 `payments.ts:106-116` 读取 `b.status`。
- **Java `savePaymentRecord`**：只插入 `t_data_fee_payment_record`，不更新 `t_data_fee_billing.status`。

此外，后端 `payments.ts:112` 将 `itemName` 映射为 `b.academic_year`，而不是费用类型名称：
```typescript
itemName: b.academic_year || '账单',  // 显示 "2025-2026" 而非 "学费"
```

**结论：学生支付后，t_data_fee_billing 表中对应费用项的 `paid_amount` 和 `status` 均未更新。教师端看到的 bills[] 仍显示未缴。**

**根因类型：后端数据同步缺失 + 字段映射错误**

---

### 问题 5：学生端支付成功后是否调用了真实支付/确认接口

**部分调用，但写入的表不对。**

调用链：
1. `store.addPaymentRecord()` → 写 localStorage + 异步调用 `api.savePaymentRecord()`（catch 忽略错误）
2. `store.savePaymentToBackend()` → `POST /api/student/payment/save-record` → Java `BillController.savePaymentRecord()`

Java 后端做了：
- `INSERT INTO t_data_fee_payment_record` ← 支付记录表
- `INSERT INTO t_data_fee_billing`（仅当该学生无 billing 记录时，创建一条 total_amount=0 的占位记录）
- **不更新** `t_data_fee_billing.paid_amount`
- **不更新** `t_data_fee_billing.status`

**结论：调用了后端接口，但接口实现不完整 — 只写支付流水表，不更新账单汇总表。**

**根因类型：后端接口实现不完整**

---

### 问题 6：支付成功后教师端列表是否重新请求接口

**是，教师端会重新请求。**

[teacher/fee-home/index.vue:252-259](teacher/fee-home/index.vue#L252-L259):
```javascript
async onShow() {
  // ...
  await this.refresh()  // 每次进入页面都会重新请求 GET /api/v1/payments/students
}
```

**但请求到的数据仍是旧的**，因为 `t_data_fee_billing.paid_amount` 从未被更新。

**结论：刷新逻辑正确，但数据源未更新。**

**根因类型：后端数据同步缺失**

---

### 问题 7：教师端状态 paidStatus 映射是否正确

**映射本身正确，但源数据错误导致状态始终为 unpaid。**

状态映射链：
```
t_data_fee_billing.paid_amount (始终为 0)
  → computeStatus(recv, 0, false, dueDate)
    → paid(0) < total(recv) 且 paid(0) === 0
      → overdue? → 'OVERDUE'
      → otherwise → 'UNPAID'
```

`STATUS_LABELS` 映射：
```typescript
paid: '已缴清', unpaid: '未缴费', partial: '部分缴费',
overdue: '已逾期', green_channel: '绿色通道'
```

前端 `fee-home` 的 `PAYMENT_KEY_STATUS_MAP`：
```javascript
unpaid: ['unpaid', 'overdue'],
partial: ['partial'],
paid: ['paid'],
green: ['channel', 'green_channel']
```

前端 `student-detail` 的 `feeTotal` computed：
```javascript
const labelMap = { paid:'已缴', unpaid:'未缴', partial:'部分', overdue:'逾期', channel:'绿通', green_channel:'绿通' }
```

**结论：映射逻辑本身无问题。问题在于后端 `computeStatus()` 收到的 `paid` 参数始终为 0（因为 billing 表未更新），导致永远返回 UNPAID/OVERDUE。**

**根因类型：后端数据同步缺失**

---

### 问题 8：后端是否只更新了学生端支付记录，没有更新教师端缴费汇总

**是的，这是核心 bug。**

Java `BillController.savePaymentRecord()` (lines 84-111):
```java
// 1. 查找或创建 billing 记录（total_amount=0 的占位行）
List<Map<String, Object>> bills = jdbc.queryForList(
    "SELECT id FROM t_data_fee_billing WHERE student_id = ... LIMIT 1", sid);
if (bills.isEmpty()) {
    jdbc.update("INSERT INTO t_data_fee_billing (..., total_amount, paid_amount, ...) " +
        "VALUES (..., 0, 0, 'PENDING', ...)");
}

// 2. 插入支付流水 ← 只做了这个
jdbc.update("INSERT INTO t_data_fee_payment_record (...) VALUES (...)");

// 3. ❌ 没有 UPDATE t_data_fee_billing SET paid_amount = paid_amount + ?, status = ...
```

**缺失的 SQL**:
```sql
UPDATE t_data_fee_billing
SET paid_amount = paid_amount + ?,
    status = CASE WHEN paid_amount + ? >= total_amount THEN 'PAID' ELSE 'PARTIAL' END
WHERE student_id = ?;
```

**结论：学生支付成功 → 只写 `t_data_fee_payment_record`（流水表） → 不更新 `t_data_fee_billing`（汇总表） → 教师端读汇总表 → 看到未缴。**

**根因类型：后端接口实现不完整（缺 UPDATE billing 逻辑）**

### 问题 9：是否存在学生端和教师端调用了两个不同数据源/API

**是的，存在多处不同数据源。**

| 维度 | 学生端 | 教师端 |
|------|--------|--------|
| JS 后端表 | `t_data_billing`（`student/bill.ts`） | `t_data_fee_billing`（`staff/payments.ts`） |
| Java 后端表 | `t_data_fee_billing`（`BillController`） | — |
| 前端价格 | 硬编码 `paySystem` | 来自 API 响应 |
| 支付状态 | localStorage | 数据库 |

**尤其是在 JS 后端中，学生端和教师端查询的是两张不同的表：**
- 学生端 JS 后端: `SELECT * FROM t_data_billing WHERE student_no = ?`
- 教师端 JS 后端: `SELECT * FROM t_data_fee_billing WHERE student_id = ?`

**即使 Java 后端正确更新了 `t_data_fee_billing`，如果学生端走 JS 后端的 `t_data_billing` 表，数据也不会一致。**

**结论：两端使用不同的数据库表 + 不同的价格来源 + 不同的状态存储。**

**根因类型：数据源架构不统一**

**是，学生端严重依赖本地缓存。**

| 页面/模块 | 数据来源 | 是否 Mock/本地 |

| 学生 `mock-api.js` `getBillList` | 返回硬编码 mockBills (totalAmount: 8480, paidAmount: 0) | **Mock** |

**根因类型：开发环境全 Mock + 生产环境接口不完整**

## 六、根因归类总结

### A. 后端数据同步缺失（最严重，问题 2/4/5/6/7/8 的根因）

**Java `BillController.savePaymentRecord()` 缺少对 `t_data_fee_billing` 的 UPDATE 操作。**

文件：[`enroll-service-java/.../BillController.java:84-111`](d:\study\backend\enroll-service-java\src\main\java\com\jfeat\enroll\controller\student\BillController.java#L84-L111)

学生支付成功后，接口只做了：
1. `INSERT INTO t_data_fee_payment_record` — 支付流水
2. `INSERT INTO t_data_fee_billing`（仅当无记录时创建占位行，`total_amount=0`）

缺失：
- `UPDATE t_data_fee_billing SET paid_amount = paid_amount + ? WHERE student_id = ?`
- `UPDATE t_data_fee_billing SET status = 'PAID'/'PARTIAL' WHERE ...`

**教师端所有查询基于 `t_data_fee_billing`，此表从未被支付流程更新 → 教师端永远显示未缴。**

### B. 前端硬编码价格（问题 1/3 的根因）

| 位置 | 硬编码值 |
|------|---------|
| 学生 `bill/index.vue` paySystem | 学费 5200 + 住宿 2000 + 教材 800 + 体检 180 + 军训 300 = **8480** |
| 学生 `checkout/index.vue` totalAmount | 同上 **8480** |
| 学生 `store.addPaymentRecord()` totalDue | 同上 **8480** |
| `businessState.js` `FEE_ITEMS_CONFIG` | 学费 5000 + 住宿 1200 + 教材 320 + 体检 200 + 军训 180 = **6900** |
| JS 后端 `payments.ts` `HARDCODED_BILLS` | 学费 5200 + 住宿 1200 + 教材 800 + 体检 180 + 军训 300 = **7680** |
| JS 后端 `student/bill.ts` `defaultBills` | 学费 5200 + 住宿 1200 + 教材 800 + 体检 180 + 军训 300 = **7680** |

**同一个"住宿费"在 4 个地方有 3 个不同值（1200 / 2000 / 0）。**

### C. 数据源架构不统一（问题 9 的根因）

```
学生端 JS 后端 ──查询──> t_data_billing       (表 A)
```
