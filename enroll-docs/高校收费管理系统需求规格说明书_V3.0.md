## 一、文档说明

### 1.1 编写目的

本文档面向高校收费管理场景，以"迎新缴费、学费管理、退费、财务报表、消息推送"五大核心业务为主线，系统化描述各功能模块的业务需求、角色权限、数据字段及业务规则。§1.4 单独给出缴费、退费、宿舍补差三类端到端闭环流程，作为系统设计与开发的强制依据。

### 1.2 适用范围

本文档适用于高校招生迎新阶段及在校期间的全周期收费管理，涵盖：

- 招生阶段预缴费（迎新缴费）
- 在校期间学费、住宿费等常规收费
- 助学金、助学贷款管理
- 退费处理
- 财务报表与数据统计
- 收费与退费消息推送
 
**不在本文档范围内：** 家长代缴场景（已移除）。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 迎新缴费 | 招生阶段新生在正式报到前完成的预缴费行为 |
| 助学贷款 | 国家为高校贫困生提供的助学贷款，含生源地贷款和校园地贷款 |
| 助学金 | 学校或政府向困难学生发放的无偿资助金 |
| 补差订单 | 因宿舍房型变更产生的费用差额订单（补缴或退款） |
| 选宿批次 | 管理员配置的在线选宿时间窗口及优先级、推送策略等参数集合 |
| 选宿优先级 | 决定学生进入在线选宿排队顺序的规则（如按缴费完成时间、随机） |
| 房型 | 宿舍床位规格（如 4 人间、6 人间），与住宿费标准绑定 |
| 床位 | 宿舍内最小分配单元；状态含正常 / 禁用 / 已占用 |
| 不住宿 | 走读生，不参与分配，不产生住宿费 |
| 规则引擎 | 约束智能分配与学生自选可用床位范围的规则集合 |
| 批扣 | 银行批量代扣，学校将应缴名单提交银行，银行统一扣款 |
| 预缴 | 学生在账单生成前提前缴纳的费用，后续自动抵扣 |
| 预缴余额 | 学生当前可用于抵扣的预缴金额，可在学生端"预缴余额"页查看 |
| 电子票据 | 支付成功后系统自动生成的电子收据，具有法律效力 |
| 支付订单 | 单次支付请求单，关联账单/明细；状态：待支付 / 支付成功 / 已关闭 / 支付失败 |
| 缴费记录 | 资金入账凭证，一条支付成功订单对应一条或多条缴费记录（按明细拆分时） |
| 合并缴费组 | 合并多笔账单一次支付时，系统记录的账单与支付金额映射关系集合 |

### 1.4 核心业务流程闭环

本节描述**缴费**、**退费**、**宿舍补差**三类端到端闭环，是各模块需求的强制基准。实现须保证：**状态机清晰、节点幂等、失败可重试或人工介入、全链路可追溯**。

#### 1.4.1 缴费闭环流程

**参与实体：** 收费标准 → 账单 / 账单明细 → 支付订单 → 缴费记录 → 电子票据 → 消息通知

**主路径（在线支付）：**

```
配置收费标准 / 个人调整
    ↓
身份绑定或学年切换 → 生成账单（状态：未缴）
    ↓
学生选择账单或明细项 → 系统按优先级规则分配金额 → 创建支付订单（待支付，30 分钟有效）
    ↓
调用第三方支付 → 用户完成付款
    ↓
支付回调（验签 + 幂等，以支付订单号为去重键）
    ↓
写入缴费记录 → 通过账单服务（分布式锁，账单 ID 粒度）串行更新已缴金额 → 重算账单状态
    ↓
生成电子票据 → 发送缴费成功通知
    ↓
（可选）更新选宿优先级 / 选宿放行条件
```

**并入同一闭环的其他入账路径：**

| 路径 | 触发 | 闭环终点 |
|------|------|----------|
| 线下收款 | 学生上传凭证 + 财务确认；或迎新现场登记 | 缴费记录 → 账单更新 → 票据 → 通知 |
| 银行批扣 | 导入银行扣款成功结果 | 同上；失败记失败明细并通知 |
| 预缴抵扣 | 支付或出账时自动扣减预缴账户 | 预缴余额减少 + 账单已缴增加 |
| 助学金抵扣 | 审批通过且发放方式为账单抵扣 | 优惠金额增加、已缴增加；无支付订单 |
| 助学贷款冲抵 | 贷款审批通过 | 同上 |
| 补差补缴 | 补差订单推送后学生在线/线下缴费 | 补差订单 → 已结清；见 §1.4.3 |

**支付订单状态机：**

| 状态 | 说明 | 可转移至 |
|------|------|----------|
| 待支付 | 已创建，等待用户付款 | 支付成功 / 已关闭（超时） / 支付失败 |
| 支付成功 | 渠道回调确认 | 终态（允许发起退费） |
| 已关闭 | 超过 30 分钟未支付，自动关闭 | 终态（可重新创建新订单） |
| 支付失败 | 渠道明确返回失败 | 终态（可重新创建新订单） |

> **支付中间态处理（D-01 已定）：** 维持两态（待支付 / 支付成功），依赖 30 分钟超时关闭。用户发起支付后切回小程序期间，账单页展示"支付订单待支付，30 分钟内有效，请在支付应用中完成付款"；超时后订单关闭，学生可重新发起。不新增"支付中"独立状态，不做前端轮询。

**部分缴费金额分配规则（D-02 已定）：**

学生选择部分缴费（缴纳金额 < 账单应缴总额）时，系统按收费项目优先级从高到低依次填满各项，直至金额用尽：

1. 按收费项目"收费优先级"字段升序排列（数值越小越优先）；
2. 当前项目"已缴金额"累加至"应缴金额"为止，剩余金额流向下一项；
3. 必缴项目（`是否必缴=1`）优先纳入，但仍按优先级排序；
4. 缴费确认页实时展示"本次分配明细预览"，学生确认后方可提交支付；
5. 任一项目已缴金额不得超过该项应缴金额。

**合并缴费与退费（D-03 已定）：**

- 合并缴费时系统创建"合并缴费组"，持久化记录各账单 ID 与本次支付中对应金额的映射；
- 退费申请时，若原缴费来自合并缴费，退费申请关联对应的合并缴费组 ID；
- 退款接口调用金额以该账单在本次合并缴费中的映射金额为上限；
- 支持对合并缴费中的单笔账单单独发起退费，其余账单不受影响。

**账单金额演算规则：**

- `已缴金额` = Σ 成功缴费记录金额 + Σ 抵扣类冲抵金额 − Σ 成功退费回退金额
- `欠费金额（展示）` = `应缴总金额` − `已缴金额` − `优惠金额`
- 账单状态：`已缴=0` → **未缴**；`0 < 已缴 < 应缴` → **部分缴**；`已缴 ≥ 应缴` → **已缴**

**并发写入保护：**

`已缴金额`字段涉及多路并发写入（支付回调、预缴抵扣、助学金冲抵、助学贷款冲抵、退费回退），所有修改必须通过账单服务的分布式锁（按账单 ID 粒度）串行执行，禁止直接 UPDATE。所有写入以"账单 ID + 来源类型 + 业务流水号"为幂等键去重。

**闭环异常与补偿：**

| 异常场景 | 处理策略 |
|----------|----------|
| 回调延迟或重复 | 以支付订单号为幂等键；重复回调不重复记账 |
| 支付成功但未生成票据 | 对账任务补生成电子票据 |
| 支付成功但账单未更新 | 对账任务按订单补写缴费记录并刷新账单 |
| 部分缴费后停止 | 账单保持部分缴；催缴模块跟进欠费项 |

#### 1.4.2 退费闭环流程

**参与实体：** 原缴费记录 / 支付订单 → 退费申请 → 退费执行记录 → 账单回退 → 电子票据处理 → 消息通知

**主路径（在线支付原路退回）：**

```
学生发起退费申请（填写退费原因）
    ↓
系统校验（见校验清单）
    ↓
创建退费申请（状态：待财务审核）
    ↓
学生端展示：「退款申请已提交，审核中」，不区分失败原因
    ↓
财务人员在职工端/PC端确认退费 → 系统调用第三方退款接口（状态：处理中）
    ↓
退款回调（验签 + 幂等）
    ↓
退费成功 → 回退账单/明细已缴金额 → 重算账单状态
    ↓
处理关联电子票据（全额退：作废原票；部分退：保留原票并记录已退金额）
    ↓
发送退费结果通知（成功/失败）
```

**退费状态机：**

| 状态 | 说明 | 学生端展示文案 |
|------|------|----------------|
| 待财务审核 | 已提交申请，财务未确认 | 退款申请已提交，审核中 |
| 处理中 | 财务已确认，已提交第三方 | 退款处理中，预计 3-5 个工作日到账 |
| 成功 | 退款到账，账单已回退 | 退款成功，已原路返回 |
| 失败 | 渠道失败，需人工介入 | 退款申请已提交，审核中（统一展示，不向学生暴露失败原因） |

> **退费失败对学生的可见程度（D-05 已定）：** 无论失败原因（系统超时、账户异常等），学生端统一展示"退款申请已提交，审核中"。失败详情仅对财务人员可见，财务人工处理后重推。学生如有疑问，可通过消息中心联系财务处。

**退费校验清单：**

| 校验项 | 不通过时行为 |
|--------|-------------|
| 原支付状态 = 支付成功 | 拒绝申请，提示"该订单不支持退费" |
| 支付日期 = 当日（在线支付） | 拒绝申请，提示"退费仅支持当日申请，如有特殊情况请联系财务处" |
| 退费金额 ≤ 可退余额 | 截断为可退上限，展示最大可退金额 |
| 同一缴费记录无重复全额退 | 拒绝申请 |
| 来自合并缴费 | 须关联合并缴费组 ID，退款金额 ≤ 该账单在组内的映射金额 |
| 涉及助学金抵扣 | 可退上限 = 原现金支付金额（不含助学金抵扣部分） |

> **退费入口展示规则：** 退费入口仅在当日支付成功的订单上**可操作**；过了当日（服务器时间所属自然日），入口变为置灰禁用状态并展示"仅支持当日申请退费"，不静默消失。前后端均以服务器时间为准，不使用客户端本地时间。

**退费申请流程（D-04 已定）：学生提交 → 财务人工确认 → 系统执行退款**

- 学生提交后状态立即变为"待财务审核"；
- 财务人员在职工端/PC端退费处理页看到该申请，确认后系统调用退款接口；
- 退款发起后状态变为"处理中"；
- 回调成功后变为"成功"，同步回退账单，发送通知；
- 退款失败时财务端展示失败原因，支持重新发起；学生端保持"退款申请已提交，审核中"展示。

**可退金额计算规则（含助学金场景）：**

- 可退金额 = min（原现金支付金额，账单已缴金额 − 优惠金额）
- 助学金抵扣部分（计入优惠金额）不可通过退费流程退回；如需退回，走助学金管理模块独立流程
- 退费申请页展示："本次最高可退金额：¥XX（不含助学金抵扣部分 ¥XX）"

**线下收费退费分支：** 不调用第三方接口；财务在后台登记退款方式（现金/转账等）并确认后，同样写入退费记录、回退账单、处理票据。

#### 1.4.3 宿舍补差闭环流程

**触发条件：** 宿舍分配结果变更导致住宿费标准变化。首次分配/选宿仅计价，不产生补差；同房型换床不产生补差。

**参与实体：** 宿舍分配记录 → 补差订单 → 账单/账单明细 → 缴费或退费闭环 → 通知

**主路径：**

```
宿舍变更（学生申请换房管理员审批通过 / 智能分配 / 管理员手动调换）
    ↓
快照原宿舍、新宿舍、原房型费用、新房型费用
差额 = 新费用 − 原费用（正数=补缴，负数=应退）
    ↓
生成补差订单（状态：待推送）
补缴截止日期 = 补差订单生成时间 + 后台配置宽限天数
    ↓
按选宿批次「补差推送策略」
    ├─ 自动推送：立即进入缴费系统
    └─ 人工确认后推送：管理员审核后触发
    ↓
推送成功 → 补差订单（已推送）
    ↓
    ├─ 差额 > 0（补缴）
    │     追加到当前学年账单明细（住宿费或补差专项，金额=差额）
    │     应缴截止日期 = 上述补缴截止日期
    │     学生端账单总览展示待缴补差 → 走缴费闭环（§1.4.1）
    │     缴清后补差订单（已结清）→ 通知「补差补缴成功」
    │
    └─ 差额 < 0（应退）
          生成独立待退金额记录（不关联原缴费记录，不受"当日退费"限制）
          财务在职工端"补差退款"列表确认后执行退款
          回退账单已缴金额 → 补差订单（已结清）→ 通知「补差退款完成」
    ↓
推送失败 → 补差订单（推送失败）→ 支持重推，幂等按补差订单号去重
```

**补差补缴截止日期（D-06 已定）：**

- 截止日期 = 补差订单生成时间 + 后台配置的宽限天数（默认值建议 7 天，管理员可在收费配置中调整）
- 选宿批次配置页新增"补差补缴宽限天数"字段（INT，必填，默认 7）
- 补差订单数据字段新增 `补缴截止日期 DATE`，在生成时写入
- 补差通知模板 `{{deadline}}` 变量绑定该字段
- 学生端待处理补差卡片展示截止日期；距截止 ≤ 3 天时以警告色高亮

**补差多次叠加规则：**

- 每次房型变更各生成一笔独立补差订单，按生成时间先后处理
- 每笔补差的"原费用"基准取**上一次已结清的住宿费用**（即当前生效房型的费用），不回溯到入学时的初始房型，确保差额链式计算
- 学生端"我的宿舍"页按时间序列展示所有调整历史及每笔补差的结清状态
- 账单总览若存在多个未结清补差订单，聚合展示：「住宿费调整待处理（共 X 笔）：待补缴 ¥XX / 待退款 ¥XX」，点击展开明细

**补差订单状态机：**

| 状态 | 说明 |
|------|------|
| 待推送 | 已计算差额，未进入缴费系统 |
| 已推送 | 已写入账单调整或生成待退记录 |
| 待补缴 | 差额 > 0，学生尚未缴清 |
| 待退款 | 差额 < 0，退款尚未完成 |
| 已结清 | 补缴或退款均已完成 |
| 推送失败 | 对接异常，可人工重推 |

**补差扩展字段（新增）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| 补差订单号 | VARCHAR(50) | 唯一，格式：DIFF{YYYYMMDD}{流水号} |
| 关联账单 ID | BIGINT | 推送后写入（补缴场景） |
| 关联缴费记录 ID | BIGINT | 补缴结清时写入 |
| 关联退费记录 ID | BIGINT | 应退结清时写入 |
| 业务状态 | 枚举 | 见上表 |
| **补缴截止日期** | DATE | 生成时计算写入（D-06） |
| 结清时间 | DATETIME | 补缴或退款完成时 |

## 二、终端架构与角色划分

### 2.1 终端概览

系统共分为四个终端，共享同一套后台数据，通过角色权限控制各终端的页面可见性与操作范围。

```
┌─────────────────────────────────────────────────────────────────┐
│                        共享后台数据层                             │
│  账单 / 缴费记录 / 退费记录 / 票据 / 助学金 / 宿舍 / 报表        │
└──────────┬──────────────┬──────────────┬──────────────┬─────────┘
           │              │              │              │
    ┌──────┴──────┐ ┌─────┴──────┐ ┌────┴──────┐ ┌────┴──────┐
    │  学生端      │ │  教师端    │ │ 财务/职工端│ │ 后台管理端│
    │ 小程序/H5   │ │ 小程序/H5  │ │ 小程序/H5 │ │  PC Web   │

## 四、模块一：迎新缴费（招生阶段预缴费）

### 3.1 功能概述

迎新缴费覆盖新生从录取通知书发放到正式报到的全过程，支持在线预缴、银行批扣等多种缴费方式，并与在线选宿联动。

**业务流程：**

```
录取通知书发放
    ↓
新生通过学号/身份证号完成身份绑定
    ↓
系统自动生成应缴账单（按学年拆分）
    ↓
（可选）在线选宿：按选宿批次与优先级选择床位 → 确认后触发住宿费写入账单
    ↓
新生选择缴费方式（在线/线下/预缴/批扣）
    ↓
缴费成功 → 生成电子票据 → 发送缴费通知
    ↓
现场报到时系统自动核验缴费状态
    ↓
（未缴费）→ 现场补缴
```

#### 3.1.1 选宿与缴费衔接配置

学校可在迎新/选宿批次中配置选宿与缴费的先后顺序及放行规则：

| 配置模式 | 说明 | 典型场景 |
|----------|------|---------|
| 先缴后选 | 必缴收费项目缴清后才开放在线选宿入口 | 确保学费等到账后再分宿 |
| 先选后缴 | 选宿与缴费并行；选宿确认后按房型生成/更新住宿费账单 | 先定房型再缴住宿费 |
| 按缴费时间排优先级 | 选宿与缴费可并行，排队序号按「必缴项缴费完成时间」升序生成 | 激励早缴费早选宿 |

> **「先缴后选」模式下的账单展示：** 学生缴清必缴项目后，支付结果页展示「温馨提示：住宿费将在您完成选宿后写入账单，请返回首页进行选宿」；首页账单总览以占位条目展示「住宿费（待选宿后生成，金额待定）」。选宿确认后系统追加住宿费到当前学年账单明细（不新建账单），并推送通知「住宿费已确认：¥XX，请尽快缴纳」。

**放行条件（可组合）：**

| 条件项 | 说明 |
|--------|------|
| 选宿批次状态 | 仅「进行中」批次允许学生选宿 |
| 必缴项缴清 | 账单中标记为必缴的项目已缴清 |
| 不住宿名单 | 走读生不参与选宿、不产生住宿费 |
| 功能开关 | 关闭「在线选宿」时仅展示预分配结果 |
| 已分配可自选 | 已被系统预分配的学生是否允许进入自选流程（批次级开关） |

#### 3.1.2 宿舍与选宿全流程（管理侧）

```
[系统管理员] 配置规则引擎、选宿批次（时间窗口/优先级/补差推送策略/补缴宽限天数）
        ↓
[宿舍负责人] 维护宿舍床位、不住宿名单
        ↓
[分配阶段，可混用]
├── 智能分配（批量 / 现场逐一）
├── 手动分配（指定 / 调换）
└── 学生在线自选（按优先级排队，受规则引擎约束）
        ↓
分配完成 → 按房型触发住宿费计费（写入账单明细）
        ↓
宿舍变更（学生申请换房/管理员调换）→ 生成补差订单 → 按策略推送缴费
```

### 3.2 收费基础配置

#### 3.2.1 收费主体管理

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 主体名称 | VARCHAR(100) | 是 | 系统内部标识，全局唯一 |
| 支付页面显示名称 | VARCHAR(100) | 是 | 学生端展示名称 |
| 支付方式 | JSON 数组 | 是 | 支持：微信、支付宝、银联、线下 |
| 排序序号 | INT | 是 | 数值越小越靠前 |
| 是否启用 | TINYINT | 是 | 0-禁用，1-启用 |
| 备注 | VARCHAR(500) | 否 | |

**业务规则：**
- 主体名称全局唯一，不允许重名
- 每个收费主体至少配置一种支付方式
- 已被收费标准引用的主体不可删除，只能禁用
- 只有启用的收费主体才在支付页面显示

#### 3.2.2 收费项目管理

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 项目名称 | VARCHAR(100) | 是 | 如：学费、住宿费、教材费 |
| 项目编码 | VARCHAR(50) | 是 | 唯一标识，创建后不可修改 |
| 收费优先级 | INT | 是 | 数值越小优先级越高；部分缴费时按此顺序分配金额 |
| 是否必缴 | TINYINT | 是 | 0-可选，1-必缴 |
| 是否启用 | TINYINT | 是 | 0-禁用，1-启用 |
| 备注 | VARCHAR(500) | 否 | |

**常见收费项目示例：**

| 项目名称 | 优先级 | 是否必缴 |
|----------|--------|---------|
| 学费 | 1 | 是 |
| 住宿费 | 2 | 否 |
| 教材费 | 3 | 是 |
| 体检费 | 4 | 否 |
| 商业保险费 | 5 | 否 |
| 军训服装费 | 6 | 否 |
| 床上用品费 | 7 | 否 |

#### 3.2.3 收费标准设置

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 标准类型 | 枚举 | 是 | 专业标准 / 个人标准 |
| 专业 ID | BIGINT | 条件必填 | 专业标准时必填 |
| 学生 ID | BIGINT | 条件必填 | 个人标准时必填 |
| 学年 | VARCHAR(20) | 是 | 如：2024-2025 |
| 收费项目 | 关联 | 是 | 关联收费项目 |
| 收费金额 | DECIMAL(10,2) | 是 | 单位：元，≥0 |
| 调整原因 | VARCHAR(500) | 条件必填 | 个人标准时必填 |

**业务规则：**
- 个人标准优先于专业标准
- 同一学生、同一学年、同一项目只取最高优先级标准
- 支持批量导入（Excel 模板），导入时验证专业、项目是否存在
- 已生成账单的收费标准不可删除，只能禁用
- 收费标准变更时自动重新计算未缴账单

### 3.3 身份绑定与账单生成

#### 3.3.1 身份绑定

**绑定规则：**
- 支持学号绑定和身份证号绑定两种方式
- 一个学生可被本人绑定（不支持家长代缴场景）
- 绑定成功后自动触发账单生成

#### 3.3.2 账单生成

**账单生成规则：**
- 绑定成功后系统自动按学年拆分生成账单
- 账单编号格式：BILL{学年}{学生 ID}{序号}
- 应缴总金额 = 各收费项目金额之和（优先个人标准，其次专业标准）
- 应缴日期默认为学年开学前 30 天
- 账单状态：未缴（已缴=0）/ 部分缴（0 < 已缴 < 应缴）/ 已缴（已缴 ≥ 应缴）

### 3.4 在线缴费

支持学生通过手机端（H5/小程序）或 PC 端完成在线缴费。

**支持的支付方式：**
- 微信支付
- 支付宝支付
- 银联支付
- 学校网财系统对接
- 银行收费系统对接

**支付流程：**

1. 选择账单 → 选择支付方式 → 预览金额分配明细（部分缴时必须展示）→ 发起支付
2. 调用第三方支付接口，生成支付订单（有效期 30 分钟）
3. 用户在第三方支付应用完成付款；切回小程序期间账单展示「支付订单待支付，30 分钟内有效」
4. 接收支付回调（验签 + 幂等），通过账单服务分布式锁更新账单状态
5. 自动生成电子票据
6. 发送缴费成功通知

**业务规则：**
- 部分缴费金额分配规则：按收费项目优先级从高到低依次填满，详见 §1.4.1
- 支持合并缴费（同时缴纳多个账单），系统记录合并缴费组及各账单金额映射，支持单账单精确退费
- 订单有效期 30 分钟，超时自动关闭；关闭后可重新创建新订单
- 支付回调幂等处理，重复回调只处理一次
- 账单已缴金额所有更新通过分布式锁串行执行

### 3.5 线下收费

财务人员在后台手动确认线下收款，支持现金、银行转账、POS 机等方式。

**业务规则：**
- 收费金额不能大于账单未缴金额，支持部分收款
- 确认收款后自动更新账单已缴金额
- 确认收款后自动生成纸质收据编号
- 作废已确认单据需管理员权限，并同步回退账单金额

### 3.6 预缴与银行批扣

#### 3.6.1 预缴管理

**业务规则：**
- 预缴金额优先用于缴纳当前学年费用，多余金额结转下一学年
- 预缴默认有效期 3 年，到期前 30 天推送到期提醒，过期自动失效
- 支持批量导入预缴数据
- 学生端"预缴余额"页可查看余额、历史预缴记录、已抵扣记录、有效期
- 首页账单总览卡片展示当前可用预缴余额，缴费时自动抵扣

#### 3.6.2 银行批扣

**业务流程：**

1. 财务人员筛选应缴学生，导出批扣文件（支持 Excel、TXT、XML 格式）
2. 提交银行执行批扣
3. 导入银行批扣结果文件
4. 系统自动更新账单状态，生成电子票据
5. 发送扣款成功/失败通知

### 3.7 学生端在线选宿（迎新阶段）

面向新生的在线选床位能力，与缴费、住宿费动态计费联动。

#### 3.7.1 功能概述

学生在选宿批次开放的时间窗口内，按系统生成的排队优先级，在规则引擎约束范围内自主选择床位。选宿**最终确认**后触发住宿费写入账单；浏览和锁定预览阶段不触发计费。

#### 3.7.2 学生端页面与操作

| 页面/操作 | 说明 |
|-----------|------|
| 选宿入口 | 首页或迎新流程中展示；显示当前状态：未开放 / 排队等待 / 可选宿 / 已选宿 / 已关闭 |
| 排队等待 | 展示本人排队序号、预计轮次；未到号时仅可浏览不可确认 |
| 选房浏览 | 按楼栋→楼层→房间→床位层级展示可选资源；支持查看宿舍布局图及床位标注 |
| 确认选宿 | 选择床位后二次确认；成功后展示分配结果并锁定床位，**触发住宿费写入账单** |
| 我的宿舍 | 随时查看已分配信息；展示变更历史和关联补差订单；申请换房入口（见 §3.7.6） |

#### 3.7.3 选宿流程

```
管理员开放选宿批次 → 系统按优先级模式生成学生排队序号
    ↓
学生登录，校验放行条件（批次进行中、非不住宿、迎新配置允许等）
    ↓
排队轮到本人 → 展示符合规则引擎的可选床位（前端展示确认时限倒计时）
    ↓
选择床位并二次确认 → 记录分配结果 → 触发住宿费写入账单
    ↓
（超时未确认）→ 释放当前床位锁定 → 排队资格处理（见 §3.7.4）
    ↓
（批次结束未选宿）→ 关闭自主选宿，由管理员手动分配或智能分配
```

**选宿锁定与断线重连：**

- 确认时限由服务端计时（锁定开始时间 + 批次配置时限），不依赖客户端时间
- 用户切后台或网络中断后重新进入：若锁定仍有效，展示剩余时间和已浏览状态（保持上下文）；若锁定已超时，展示超时说明并按排队资格规则处理
- 前端确认时限倒计时：剩余 2 分钟时红色高亮提醒，剩余 30 秒时弹出强提示弹窗

#### 3.7.4 选宿超时后的排队资格处理（D-07 已定）

选宿超时（锁定期内未完成确认）后，**取消本次资格，不再维护排队状态，直接进入剩余可用床位的自由选择**：

- 超时后学生重新进入选宿入口，展示当前所有**剩余未分配床位**，可直接选择（不再排队等号）
- 剩余床位展示仍受规则引擎约束（性别、专业、学院等规则继续生效）
- 批次结束后不再允许自主选宿，由管理员手动或智能分配兜底

#### 3.7.5 与缴费的联动规则

| 规则 | 说明 |
|------|------|
| 放行条件 | 可配置：必缴项缴清后方可进入选宿；或并行但优先级按「缴费完成时间」排序 |
| 住宿费计费时机 | **选宿最终确认后**触发，写入当前学年账单明细；超时释放不触发计费；若计费已触发后超时则自动回滚 |
| 房型变更 | 学生申请换房审批通过或管理员调换后，自动生成补差订单并推送缴费（见 §1.4.3） |
| 功能开关 | 学校关闭「在线选宿」时，学生端不展示选宿入口，仅支持预分配结果查询 |

#### 3.7.6 学生申请换房（D-08 已定）

学生可在选宿批次进行中、且"已分配学生可自选"开关为开时，主动发起换房申请：

**申请流程：**

```
学生在「我的宿舍」页点击「申请换房」→ 填写换房原因 → 提交申请
    ↓
宿舍负责人在后台「换房申请审批」列表查看 → 审批（通过/拒绝）
    ↓
审批通过 → 学生重新进入选宿流程（浏览剩余床位并确认）→ 旧床位同步释放
         → 触发补差计算 → 生成补差订单（若房型变更）→ 通知学生
    ↓
审批拒绝 → 发送拒绝通知 → 学生保持原宿舍不变
```

**约束：**
- 批次已结束后不允许通过此入口发起换房申请
- "已分配学生可自选"开关为关时，入口不展示
- 换房申请审批操作日志不可删除

#### 3.7.7 智能提示

当某房型已选人数未达到楼栋开放阈值时，选房页展示提示（如「当前 4 人间选择人数较少，该楼栋可能无法开放」）。阈值类型与数值在选宿批次中配置（固定人数或床位占用率百分比）。

#### 3.7.8 约束

- 仅展示、操作本人数据；性别、专业、学院等规则由规则引擎强制校验，前端不可绕过
- 禁用床位不可选；已被他人占用的床位实时不可选（并发控制，防重复选床）
- 选宿批次「进行中」时，批次级配置锁定；学生端只读已结束批次的历史结果

### 3.8 不住宿管理

走读生不参与床位分配，不产生住宿费。

| 功能点 | 说明 |
|--------|------|
| 手动录入 | 逐条维护不住宿学生 |
| 批量导入 | Excel 模板导入 |
| 查询导出 | 按学院、专业、班级筛选 |
| 转为住宿 | 从不住宿改为住宿后，须重新进入分配/选宿流程 |

**学生端申请住宿：** 当学生当前状态为「不住宿」时，「我的宿舍」页展示「申请住宿」入口。学生提交申请原因后，宿舍负责人在后台审批；审批通过后学生进入选宿/分配流程并产生住宿费账单。若不提供自助入口，「我的宿舍」页须展示明确的联系方式指引，不得空白。

### 3.9 宿舍信息管理（后台）

#### 3.9.1 宿舍信息管理

**床位属性字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| 楼栋 | VARCHAR(50) | 宿舍楼名称/编号 |
| 楼层 | INT | 所在楼层 |
| 房间号 | VARCHAR(20) | 房间编号 |
| 床位号 | VARCHAR(20) | 铺位编号 |
| 房型 | 枚举 | 如 4 人间、6 人间；与住宿费标准绑定 |
| 性别限制 | 枚举 | 男/女/不限 |
| 所属学院 | VARCHAR(100) | 可选，用于预分配范围 |
| 床位状态 | 枚举 | 正常/禁用/已占用 |
| 布局图 | 文件 | 图片；含床位坐标 JSON |

#### 3.9.2 规则引擎

**角色：** 系统管理员。规则同时作用于智能分配与学生自选，选宿批次「进行中」后规则锁定不可改。

| 规则类型 | 主体 | 目标 | 示例 |
|----------|------|------|------|
| 专业-房型 | 专业 | 可选房型 | 护理专业仅 4 人间 |
| 性别-楼栋 | 性别 | 可选楼栋 | 女生仅 A、B 栋 |
| 学院/班级-楼层 | 学院或班级 | 楼层范围 | 计院 3–5 层 |
| 后台房间号 | 班级 | 指定房间 | 301 室归班级 A |

**优先级（高→低）：** 性别限制 → 专业/学院规则 → 班级规则。冲突时高优先级覆盖低优先级。

#### 3.9.3 选宿批次

**生命周期：** `未开始`（可改配置）→ `进行中`（配置锁定）→ `已结束`（只读）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 批次名称 | VARCHAR(100) | 是 | |
| 适用年级 | INT | 是 | |
| 选宿开始时间 | DATETIME | 是 | |
| 选宿结束时间 | DATETIME | 是 | |
| 优先级模式 | 枚举 | 是 | 缴费时间顺序 / 随机 |
| 随机种子 | BIGINT | 否 | 随机模式；null=不固定 |
| 智能提示阈值类型 | 枚举 | 否 | 固定人数 / 床位占用率% |
| 智能提示阈值值 | DECIMAL | 否 | |
| 已分配学生可自选 | TINYINT | 是 | 预分配后是否允许再自选 |
| 补差推送策略 | 枚举 | 是 | 自动推送 / 人工确认后推送 |
| **补差补缴宽限天数** | INT | 是 | 默认 7，用于计算补缴截止日期 |
| 选宿与缴费模式 | 枚举 | 是 | 先缴后选 / 先选后缴 / 并行按缴费时间排优 |
| 轮到后确认时限 | INT | 否 | 分钟；超时释放选房锁定 |
| 状态 | 枚举 | 系统 | 未开始/进行中/已结束 |

**业务规则：**
- 同一年级同一时间仅允许一个「进行中」批次
- 开放批次时按优先级模式生成全体学生排队序号
- 关闭批次后，未选宿学生由管理员手动分配或触发智能分配
- 随机种子固定时，相同种子可复现排队顺序

#### 3.9.4 床位分配

**智能分配：**

逻辑：过滤不住宿 → 规则引擎筛可用床位 → 匹配房型/楼层 → 写分配记录与操作日志 → 触发住宿费计费。

**手动分配：**

| 功能点 | 说明 |
|--------|------|
| 手动指定 | 指定学生到指定床位 |
| 调换床位 | 两名学生互换床位 |
| 操作日志 | 操作人、时间、变更前后；不可删除 |

调换或变更房型后自动生成补差订单（见 §1.4.3）。

#### 3.9.5 约束与边界

| 场景 | 约束 |
|------|------|
| 性别 | 任何分配方式均不得跨性别；系统强制校验 |
| 禁用床位 | 不参与智能、手动、自选 |
| 选宿时间窗 | 超窗后不可自主选宿；超时可直接选剩余床位（无需排队） |
| 规则/批次锁定 | 批次进行中不可改规则与批次参数 |
| 不住宿 | 不出现在分配与选宿流程，不计住宿费 |
| 补差订单 | 同学期多次调整各生成独立订单；差额基准取上一次已结清的住宿费用 |
| 并发选床 | 同一床位仅允许一名学生确认成功；高并发下防重复占用 |

### 3.10 迎新现场缴费核验

现场报到时，工作人员实时查看学生缴费状态，决定是否放行。

**核验规则：**
- 已缴清：直接放行
- 未缴清：提示补缴

**现场统计：**
- 实时展示新生缴费数据（已缴人数、未缴人数、缴费金额）
- 支持按学院、专业、班级维度统计缴费完成率

### 4.2 宿舍动态计费

根据宿舍房型动态计算住宿费，宿舍调整时自动生成补差订单（见 §1.4.3）。

**计费规则：**
- 费用与房型绑定，不同房型对应不同住宿费标准（元/学年）
- 费用标准按年级隔离，支持设置生效日期，历史标准保留不删除
- 不住宿学生不产生任何计费记录
- 住宿费**仅在选宿最终确认**或**管理员分配完成**时写入账单，预览/锁定阶段不写入

### 4.3 助学金管理

#### 4.3.1 助学金类型

| 类型 | 说明 |
|------|------|
| 国家助学金 | 国家面向困难学生发放 |
| 学校助学金 | 学校自设项目 |
| 社会助学金 | 企业或社会机构捐助 |
| 临时困难补助 | 针对突发困难学生的一次性补助 |

#### 4.3.2 助学金申请流程

```
学生提交申请（填写家庭情况、上传证明材料）
    ↓
辅导员/班主任初审（待确认：Q1 是否需要多级审批，暂按此流程实现）
    ↓
学院负责人复审
    ↓
学工处审批
    ↓
审批通过 → 发放助学金
    ↓

#### 4.3.4 助学金与账单联动

- 冲抵记录保留完整审计日志

#### 4.3.5 业务规则

- 审批通过后不可修改申请金额，只能由审批人调整发放金额
- 助学金发放记录不可删除，只能作废（需管理员权限）

## 六、模块三：退费管理

### 5.1 功能概述

支持学生申请退费，财务人员确认后系统执行退款，并同步更新账单状态。当前版本不支持跨日退费，退费申请须当日发起。

### 5.2 退费场景

| 场景 | 说明 |
|------|------|
| 在线支付退费 | 当日内对在线支付成功的订单发起退款，原路返回 |
| 线下收费退费 | 财务人员手动处理线下收款的退费，记录退款方式 |
| 宿舍补差退款 | 因宿舍房型变更差额为负时，财务在职工端确认后执行退款（独立待退记录，不关联原缴费记录，不受当日限制） |

> 助学金冲抵退回不走退费流程，通过助学金管理模块独立处理。

### 5.3 退费流程

```
学生发起退费申请（填写退费原因）
    ↓
系统校验（原支付成功、支付当日内、退费金额 ≤ 可退余额、非重复全额退）
    ↓
创建退费申请（状态：待财务审核）
    ↓
学生端展示：「退款申请已提交，审核中」
    ↓
财务人员在职工端/PC端查看申请，点击确认退费
    ↓
系统调用第三方退款接口（状态：处理中）
    ↓
退款回调（验签 + 幂等）
    ↓
退费成功 → 通过账单服务分布式锁回退已缴金额 → 重算账单状态
    ↓
处理关联电子票据（全额退作废；部分退保留原票并记录已退金额）
    ↓
发送退费结果通知
    ↓
（若来自补差应退）更新补差订单 → 已结清
```

### 5.4 退费数据字段

**退费申请：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 申请编号 | VARCHAR(50) | 是 | 格式：RFA{YYYYMMDD}{6 位流水号} |
| 原缴费记录 ID | BIGINT | 是 | |
| 合并缴费组 ID | BIGINT | 否 | 来自合并缴费时必填 |
| 学生 ID | BIGINT | 是 | |
| 账单 ID | BIGINT | 是 | |
| 退费金额 | DECIMAL(10,2) | 是 | ≤ 可退余额（= 原支付金额 − 优惠金额部分）|
| 退费原因 | VARCHAR(500) | 是 | |
| 申请状态 | 枚举 | 是 | 待财务审核/处理中/成功/失败 |
| 申请人 ID | BIGINT | 是 | |

**退费记录：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 退费单号 | VARCHAR(50) | 是 | 格式：REF{YYYYMMDD}{6 位流水号} |
| 原订单编号 | VARCHAR(50) | 是 | |
| 退费金额 | DECIMAL(10,2) | 是 | |
| 原支付方式 | VARCHAR(20) | 是 | |
| 退款账户 | VARCHAR(100) | 否 | |
| 退费状态 | 枚举 | 是 | 待财务审核/处理中/成功/失败 |
| 退费交易流水号 | VARCHAR(100) | 否 | 第三方退款流水号 |
| 退款时间 | DATETIME | 否 | |
| 失败原因 | VARCHAR(500) | 否 | 仅财务侧可见 |

### 5.5 业务规则

**退费条件：**
- 只能对支付成功的订单进行退费
- 在线支付退费须在支付当日内（服务器时间所属自然日）发起
- 退费入口过了当日置灰并提示原因，不静默消失
- 已退款的订单不能再次全额退费（支持部分退后再退剩余）

**退费金额规则：**
- 可退上限 = min（原现金支付金额，账单已缴金额 − 优惠金额）
- 助学金抵扣部分（计入优惠金额）不参与退费上限计算
- 退费申请页展示「本次最高可退金额：¥XX（不含助学金抵扣部分 ¥XX）」
- 合并缴费场景：退费金额 ≤ 该账单在合并缴费组内的映射金额

**退费处理流程（D-04）：**
- 学生提交申请后进入「待财务审核」状态
- 财务人员在职工端退费处理页手动确认，系统发起退款
- 退款回调成功后状态变为「成功」，账单联动回退
- 退款失败时财务端显示失败原因，支持重新发起；学生端统一显示「退款申请已提交，审核中」（D-05）

**账单回退规则：**
- 退费成功后通过账单服务分布式锁串行回退已缴金额
- 账单状态根据已缴金额重新计算

**失败处理：**
- 退费失败原因仅对财务人员可见；学生端统一显示「退款申请已提交，审核中」
- 财务人员在职工端可查看失败原因并重新发起
- 失败记录保留完整审计日志

### 5.6 退费查询

| 角色 | 可查看范围 |
|------|----------|
| 系统管理员/财务人员 | 所有退费记录 |
| 教师/辅导员 | 本班学生退费记录 |
| 学生 | 个人退费记录 |

## 十二、非功能性需求

### 11.1 性能要求

| 场景 | 性能指标 |
|------|----------|
| 在线缴费 | 支付发起响应时间 < 3 秒 |
| 账单查询 | 查询响应时间 < 1 秒 |
| 批量账单生成 | 支持 10000 人以上批量生成，异步处理 |
| 报表统计 | 常规报表响应时间 < 5 秒 |
| 并发支付 | 支持迎新高峰期 500+ 并发支付 |
| 账单写入 | 分布式锁保证串行；单账单锁等待超时 ≤ 3 秒，超时返回重试 |

### 11.2 安全要求

- 支付渠道配置（商户号、密钥等）加密存储
- 支付回调需验证签名，防止伪造
- 学生身份信息（身份证号等）脱敏展示
- 所有操作记录审计日志，不可删除
- 退费操作须财务二次确认，防止误操作

### 11.3 可用性要求

- 系统可用性 ≥ 99.9%
- 迎新高峰期（每年 9 月）提前做好容量规划
- 支付回调处理需保证幂等性

### 11.4 兼容性要求

- 管理端：支持主流浏览器（Chrome、Firefox、Edge）
- 学生端：支持 PC 端和手机端（H5/微信小程序）
- 移动端适配：响应式设计，支持 iOS 和 Android
